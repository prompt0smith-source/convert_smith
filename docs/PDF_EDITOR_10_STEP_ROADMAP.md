# PDF Editor 10-Step Implementation Roadmap

이 문서는 Convert Smith PDF 편집기를 Acrobat Pro에 가까운 "원형 보존형 편집기"로 만들기 위한 단계별 작업 기준입니다.

목표는 PDF를 웹 폼처럼 다시 그리는 것이 아니라, 원본 PDF 렌더링을 기준으로 텍스트, 이미지, 선, 표를 각각 인식하고 사용자가 직접 선택, 이동, 수정, 삭제, 저장할 수 있게 만드는 것입니다.

## 핵심 원칙

- 원본 PDF 페이지의 보이는 형태를 우선 보장합니다.
- 수정 모드가 아닐 때는 PDF 뷰어처럼 텍스트 드래그 선택이 가능해야 합니다.
- 수정 모드에서는 선택한 텍스트, 이미지, 선, 표만 편집 오버레이로 올라와야 합니다.
- 저장 전까지 원본 파일은 변경하지 않습니다.
- 저장은 새 파일 또는 명시한 출력 파일로만 수행합니다.
- 글꼴을 임의 대체하지 않고, PDF 내 글꼴 정보와 사용자의 로컬 글꼴을 최대한 매칭합니다.
- 인식 실패 시 깨진 결과를 저장하지 않고 명확한 오류와 디버그 로그를 남깁니다.

## 1. 엔진 경계 고정

작업:

- PDF 편집기를 기존 PDF 도구, 파일 변환 모드와 분리합니다.
- Viewer window, editor model, save engine, debug log service를 서로 독립된 파일로 유지합니다.
- 한 파일이 과도하게 커지지 않도록 renderer, main service, type, model을 분리합니다.

완료 기준:

- PDF 편집기 진입 경로가 명확합니다.
- 기존 PDF 병합/분할/변환 기능과 상태가 섞이지 않습니다.
- 디버그 로그가 편집기 렌더링, 텍스트 추출, 저장 실패를 구분해서 기록합니다.

## 2. 원본 렌더링 기준선 확립

작업:

- PDF 페이지는 Chromium/PDFium 계열 원본 렌더링을 기준으로 표시합니다.
- fallback renderer는 원본과 비교해 글꼴, 자간, 줄간격이 틀어지는 경우 편집 저장에 사용하지 않습니다.
- 로딩 화면은 Convert Smith의 기존 물방울 로딩 UI와 진행 바를 사용합니다.

완료 기준:

- 같은 PDF를 Chrome/Acrobat과 비교했을 때 기본 보기에서 글꼴과 간격이 눈에 띄게 깨지지 않습니다.
- 렌더 실패 시 "자체 Viewer 렌더링 실패" 로그에 원인과 파일 정보가 남습니다.

## 3. 텍스트 추출 안정화

작업:

- PDF.js textContent, ToUnicode, 글리프 위치, transform matrix, direction, baseline 정보를 정규화합니다.
- 한국어/영어는 좌->우, 위->아래 읽기 순서를 기본으로 정렬합니다.
- 인코딩이 깨지는 CID/embedded font 케이스는 대체 파서를 추가로 시도합니다.

완료 기준:

- 선택 전에도 문자열이 네모나 깨진 문자로 보이지 않습니다.
- 수정 모드 진입 전후의 텍스트 위치와 간격이 달라지지 않습니다.
- 드래그 선택 텍스트와 편집 대상 텍스트가 같은 문자열을 가리킵니다.

## 4. 로컬 글꼴 매칭

작업:

- PDF 내부 font name, base font, embedded subset name을 추출합니다.
- Windows 로컬 글꼴 목록과 family/style/weight 기준으로 매칭합니다.
- 매칭 실패 시 문서 기본 대체 글꼴을 사용하되 UI에 표시합니다.

완료 기준:

- 로컬에 있는 글꼴은 수정 전후에 동일 계열로 표시됩니다.
- 한 문서에 글꼴이 두 개 이상 있어도 각 텍스트 박스가 자신의 글꼴 후보를 유지합니다.
- 저장 후 다시 열어도 수정된 텍스트의 글꼴이 임의로 바뀌지 않습니다.

## 5. 인라인 텍스트 편집 모델

작업:

- 하단 별도 텍스트 수정 패널을 제거하고, PDF 페이지 위에서 직접 수정되는 것처럼 보이게 합니다.
- 텍스트 박스는 클릭으로 선택하고, 더블 클릭 또는 수정 모드에서 바로 입력할 수 있게 합니다.
- 입력 중에는 `dir="ltr"`, `unicode-bidi: plaintext`, `writing-mode: horizontal-tb`를 고정합니다.
- 빈 배경 클릭 시 현재 편집 값을 커밋합니다.

완료 기준:

- 입력 커서가 텍스트 영역에서는 I-beam으로 보입니다.
- 우측을 클릭해도 입력 방향이 우->좌로 뒤집히지 않습니다.
- 텍스트가 칸 내부에서 잘리지 않고 최소 높이와 폭을 자동 보정합니다.

## 6. 저장 파이프라인

작업:

- 원본 텍스트를 단순히 위에 덮어쓰는 방식이 아닌, 선택 영역을 안정적으로 마스킹한 뒤 새 텍스트를 삽입합니다.
- 배경 선과 표를 가리지 않도록 텍스트 bounding box와 baseline을 계산합니다.
- 저장 전후 렌더 비교를 수행하고 큰 차이가 있으면 실패 처리합니다.

완료 기준:

- 수정 전 텍스트가 새 텍스트 뒤에 겹쳐 보이지 않습니다.
- 흰 배경 마스킹이 표 선이나 주변 요소를 침범하지 않습니다.
- 저장 후 다시 열어도 수정값이 되돌아가지 않습니다.

## 7. 이미지 객체 인식 및 편집

작업:

- PDF 내부 이미지 XObject 위치, 크기, 회전, 투명도를 추출합니다.
- 이미지는 선택, 이동, 삭제, 교체를 지원합니다.
- 이미지 원본 추출이 불가능한 경우 렌더 기준 객체로 표시하되 저장 기능은 제한합니다.

완료 기준:

- 로고, 서명, 스탬프 이미지를 텍스트와 분리해서 선택할 수 있습니다.
- 이미지를 이동해도 텍스트 레이어가 같이 움직이지 않습니다.
- 삭제/교체 시 결과 PDF 검증을 통과합니다.

## 8. 선과 표 구조 인식

작업:

- path stroke, thin rectangle, 반복 선을 선 객체로 인식합니다.
- 수평/수직 선 교차를 기준으로 표 후보를 생성합니다.
- 표 셀 내부 텍스트와 표 선을 논리적으로 연결합니다.

완료 기준:

- 표는 단순 이미지가 아니라 선/셀 후보로 하이라이트됩니다.
- 표 선 삭제, 이동, 셀 텍스트 선택이 서로 충돌하지 않습니다.
- 표 내부 텍스트 수정 후 셀 범위를 넘어가면 박스가 확장되거나 경고합니다.

## 9. 편집 UX와 단축키

작업:

- Ctrl+Z, Ctrl+Y, Ctrl+S를 PDF 편집기에도 적용합니다.
- Delete는 선택 객체 삭제로 동작하고, 삭제 확인 말풍선은 선택 객체를 가리지 않게 배치합니다.
- 수정 모드가 아닐 때는 일반 PDF 뷰어처럼 텍스트 블록 선택이 가능해야 합니다.

완료 기준:

- 텍스트, 이미지, 선, 표 객체가 같은 선택 규칙을 따릅니다.
- 삭제 버튼과 Delete 키가 같은 삭제 모델을 사용합니다.
- Undo/redo 후 화면과 저장 모델이 일치합니다.

## 10. 회귀 테스트와 사용자 검증

작업:

- 대표 PDF 샘플을 기준으로 렌더, 추출, 편집, 저장, 재열기 테스트를 자동화합니다.
- 변환/병합/분할 기능과 PDF 편집기 상태가 서로 오염되지 않는지 확인합니다.
- 사용자 검증 체크리스트를 릴리스 전 단계로 고정합니다.

완료 기준:

- `npm run build`와 `npm run smoke`가 통과합니다.
- 편집 저장 후 원본 파일은 변경되지 않습니다.
- 실패 시 결과 파일을 성공으로 표시하지 않습니다.

## 현재 구현 위치

현재 Convert Smith는 PDF 편집기 별도 진입, 원본 뷰어 창, 텍스트 편집 오버레이, 일부 글꼴 매칭, 디버그 로그, 이미지/선/표 후보 인식의 기반을 갖고 있습니다.

아직 Acrobat Pro 수준의 완전한 객체 편집 엔진은 아닙니다. 특히 이미지 객체 저장 편집, 표 구조 편집, 모든 embedded/subset font의 완전한 보존, 저장 전후 자동 시각 비교는 단계적으로 강화해야 합니다.
